Model Treeview

The model treeview is located on the left top of the phase application below the Main Toolbar. As soon as Phase is loaded, this treeview is populated with all Pentaho Analysis Models, or Mondrian Schemas which are available in the Pentaho Business Analytics Server. The main purpose of the model treeview is to offer a way to navigate through the available models, and to select a model, or an item within a model, to work with.

Model Treeview Nodes

The nodes in the model treeview reflect the structure of the available mondrian schemas. The structure of the treeview is close to the actual physical document structure of the mondrian schemas. However, the model treeview tries to simplify and abstract from the actual schema document structure where it makes sense.

In particular, the model treeview only shows the logical structure of the mondrian schemas. In other words, the model treeview is mainly reflects the dimensional model defined by the mondrian schema. The model treeview does not show any of the relational entities that are used to define these OLAP entities.

Interacting with Model Treeview nodes

Nodes in the treeview have several hotspots which you can click:

  • Clicking the + or - button (if present) just left of the node's icon, will change the state of the node. This allows you to inspect and navigate mondrian models.
  • Clicking the label (name) of the treenode, just to the right side of the node's icon will select that node (and it's corresponding model or model element)

Navigating and Inspecting schemas

A node in the model treeview is either a leaf node or not. Leaf nodes do not have childnodes, whereas non-leaf nodes can and often will have childnodes.

Non-leaf nodes can be either expanded or collapsed.

Collapsed nodes have a + toggle button in front of their icon. If a collapsed node has child nodes, then these will not be visible as long as the node is in a collapsed state. Clicking the + button of a collapsed node will change the node to an expanded state.

Expanded nodes reveal their child node if present, and they will appear slightly indented beneath their parent node. Expanded nodes have a - button in front of their icon. Clicking the - button will hide the nodes childnodes (if any) and set the node to a collapsed state.

So, by expanding and collapsing nodes, the structure of all available mondrian schemas can be revealed and inspected.

Selecting a model or a model element

Clicking on the label (name) of the node, just right of its icon will select the corresponding modelelement. The selected treenode will change its appearance to have a blue background and white text. As a response to selecting a node in the model tree node, an editor that is appropriate for that type of model element will load in the editor area.

Dragging and Dropping Treenodes

You can drag nodes in the model treeview to change their ordinal position, or to import or move them to other parent model elements. Such an action will have the effect of actually moving the corresponding model element within the schema.

In a few cases, such as with the levels within a hierarchy, the order of the schema elements does matter. But generally the ordinal position of model elements is of no consequence with regard to schema functionality. However, a custom order may be preferred by the schema author.

Node types

Nodes in the model treeview are of a particular type and represent either a mondrian schema or an element within such a schema. The nodes have both a label and an icon. The icon indicates the type of modelelement that is represented by the node. The name indicates the identifiers (the name) of the model element that the node represents.

The following list explains the different node types:

Schema node
A schema node represents a Pentaho Analysis Model - (a Mondrian Schema).
Cube node
A cube node represents a Cube. A cube is a domain-specific collection of measures and corresponding dimensions. Cube nodes can be found beneath schema nodes.
Shared Dimension node
A shared dimension node represents a shared dimension. A shared dimension is essentially a so-called conformed dimension which can be used by multiple cubes. Shared dimension nodes can be found beneath schema nodes.
Private Dimension node
A private dimension node represents a private dimension. A private dimension is a dimension that is specific to just one particular cube. Shared dimension nodes can be found beneath cube nodes.
Dimension Usage node
A dimension usage node represents a dimension usage. A dimension usage is essentially a role playing dimension: it refers to a shared dimension but is given a specific role and boud to a specific context within a cube. Dimension usage nodes can be found beneath cube nodes.
Hierarchy node
A hierarchy node represents a hierarchy within a shared or private dimension.
Level node
A level node represents a leve within a hierarchy.
Measure node
A measure node represents a measure. Measure nodes can be found beneath cube nodes. Measure nodes are leaf nodes.
Calculated Member Node
A calculated member node represents a calculated member. Calculated member nodes can be found beneath cube nodes and virtual cube nodes. Calculated member nodes are leaf nodes.
